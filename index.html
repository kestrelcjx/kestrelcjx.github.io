再论向上转型

转机

将一个方法调用同一个方法主体关联起来称为绑定。若在程序执行前绑定，叫做前期绑定；
若在运行时根据对象的类型进行绑定，成为后期绑定（动态绑定或运行时绑定）。
Java中除了static方法和final方法（private方法属于final方法）之外，
其他方法都是动态绑定的。

构造器与多态

  • 基类的构造器总是在导出类的构造过程中被调用，而且按照继承层次逐渐向上链接，
    以使每个基类的构造器都得到调用。在导出类的构造器主体中，如果没有明确指定调用
    某个基类构造器，它就会默认调用基类默认无参构造器，如果不存在默认构造器，
    编译器会报错，则需要显式调用一个带参数的基类构造器。
  • 继承与清理
    当覆盖被继承的清理方法时，务必记住调用基类版本的清理方法，避免未清理基类垃圾。

协变返回类型

Java SE5中添加了协变返回类型，它表示在导出类中的被覆盖方法可以返回基类方法的
返回类型的导出类型，在java早期版本只能具有相同返回类型。

final关键字

  • final数据
    final标识基本类型，表示常量，不可修改值；
    final标识对象，引用对象不可变，但可以改变对象本身的值。
    空白final，指声明为final但不给出初值，那么必须在构造器中赋初值。
    final参数，意味着在方法中无法更改参数引用所指向的对象。
  • final方法
    final修饰方法，表示把方法锁住，以防任何继承类修改它的含义，确保在继承中是方法
    行为保持不变，并且不会被覆盖。但是，“覆盖”只有在某方法是基类的接口的一部分时
    才会出现。即，必须能够将一个对象向上转型为它的基本类型并调用相同方法。如果某方法为private，
    它就不是基类的接口的一部分。此时如果导出类具有相同名称但未public、protected或
    包范文权限方法的方法，则并不会覆盖该方法，仅是生成了一个新方法而已。
  • final类
    final修饰类，表示该类不能被继承。

包：库单元

如果未设置源文件所属包，则会位于未命名包（默认包）中。
当编写一个Java源代码文件时，此文件通常被称为编译单元。
每个编译单元都必须有一个后缀.java，而在编译单元内则有一个public类，
该类的名称必须与文件的名称相同（包括大小写，但不包括后缀名.java）。
每个编译单元只能由一个pubic类（也可以没有）。
创建包时，按照惯例，package名称第一部分是类创建者的反顺序的Internet域名
（例如：com.kestrelcjx.xxx）。package名称将分解为机器上的一个目录。
Java解释器的运行过程如下：首先，找出环境变量CLASSPATH。CLASSPATH包含一个或
多个目录，用作.class文件的根目录。从根目录开始，解释器获取包的名称并将
每个句号替换为反斜杠，以从CLASSPATH根肿产生一个路径名称。得到的路径会与CLASSPATH路径
中的各个不同项相连接，解释器就在这些目录中查找与你所创建类的名称相关的.class文件。

初始化与清理

用构造器确保初始化

方法重载

  • 区分重载方法：重载方法名称相同，参数数量、类型或顺序不同，但无法以返回值不同实现重载。
  • 涉及基本类型的方法重载：基本类型从一个“较小”的类型自动提升至一个“较大”的类型。
    char略有不同，如果没有char参数的方法，将直接提升至int。

默认构造器

如果不提供构造器，将自动创建一个无参数的默认构造器，如果提供了构造器，则不会自动创建。

this关键字

  • 使用对象调用方法，编译器会暗自将“所操作对象的引用”作为第一个参数传递给方法。
  • 使用this关键字在方法内部获得当前对象的引用，但只能在方法内部使用。
  • 在构造器中可以使用this()调用重载的其他构造器。
  • static方法中，不能使用this，且不能调用非静态方法。

清理：终结处理和垃圾回收

Java中的对象创建在堆中，对象不再使用时必须有垃圾回收器回收所占用内存。
发生垃圾回收时，会使用对象的finalize()方法（Object类中的protected方法）。
垃圾回收一边回收空间，一边使堆中的对象紧凑排列。

  • 对象可能不被垃圾回收
  • 垃圾回收并不等于析构
  • 垃圾回收只与内存有关

垃圾回收机制：

  1. 引用记数是一种简单但速度很慢的垃圾回收技术。每个对象都含有一个引用记数器，
    当有引用连接至对象时，引用记数加1。当离开作用域或被设置为null时，引用记数减1。
    垃圾回收器会在含有全部对象的列表上遍历，当发现某个对象引用记数为0时，就释放
    其所占用的空间（但是，引用记数模式通常会在记数值为0时立即释放对象）。引用记数
    常用来说明垃圾收集的工作方式，但似乎从未被应用于如何一种Java虚拟机实现中。
  2. 另一种垃圾回收：从堆栈和静态存储区开始，遍历所有的引用，找到所有“活”的对象
    。对于发现的每个引用，必须追踪它所引用的对象，然后是此对象包含的所有引用，直到
    “根源于堆栈和静态存储区的引用”所形成得到网络全部被访问为止。在这种方式下，Java虚拟机
    将采用一种自适应的垃圾回收技术，至于如何处理找到的存活对象，取决于不同的Java虚拟机实现。
    （例如：“停止-复制”、“标记-清扫”）

构造器初始化

  • 在类的内部，变量定义的先后顺序决定了初始化的顺序。即使变量定义散步于方法定义之间，
    它们仍旧会在方法（包括构造器）被调用之前得到初始化。
  • 静态初始化只有在必要时刻才会进行。如果不创建对象，也不引用对象的静态变量，那么不会进行初始化。
    初始化顺序是先静态对象，而后是非静态对象。
  • 显示的静态初始化（静态块）
  • 非静态实例化初始化
    与静态初始化子句相似，但少了static关键字，它保证无论调用哪个显示构造器，都会执行其中的初始化。

总结一下对象的创建过程，假设有一个名为Dog的类：

    的对象时（构造器可以看成是静态方法），或者Dog类静态方法/静态域首次被访问时，Java解释器必须查找类路径，

  4. 这块存储空间会被清零，这就自动地将Dog对象中的所有基本类型数据都设置为默认值，而引用类型为null。
  5. 执行所有出现于字段处的初始化动作。
  6. 执行构造器。

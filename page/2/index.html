擦除的神秘之处

Java中使用擦除实现泛型。在泛型代码内部，无法获得任何有关泛型参数类型的信息。
擦除的代价是显著的。泛型不能用于显式地引用运行时类型的操作之中，例如转型、
instanceof操作和new表达式，因为所有类型信息都丢失了。

擦除的补偿

无法在泛型中使用instanceof，如果引入类型标签，就可以转而使用动态的isInstance()。
编译器将确保类型标签可以匹配泛型参数。

边界

通配符

  • 可以向导出类型的数组赋予基本类型的数组引用。

  • 与数组不同，泛型没有内建的协变类型。这是因为数组在语言中是完全定义的，
    因此可以内建了编译器和运行时的检查，但是在使用泛型时，编译器和运行时
    系统都不知道你想用类型做什么，以及应该采用什么样的规则。但是，有时你
    想要在两个类型之间建立某种类型的向上转型关系，这正是通配符所允许的。

  • 当你指定一个ArrayList<? extends Fruit>时，add()的参数就变成了“? exntends Fruit”。
    从这个描述中，编译器并不能了解这里需要Fruit的哪个具体子类型，因此它不会接受任何类型
    的Fruit。
    在使用contains()和indexOf()时，参数类型是Object，因此不会涉及任何通配符，而编译器
    也将允许这个调用。

  • 逆变。可以声明通配符是由某个特定类的任务基类来界定的，方法是指定<? super MyClass>，
    甚至或者使用类型参数<? super T>（尽管你不能对泛型参数给出一个超类型边界；即不能声明
    ）。这使得你可以安全地传递一个类型到泛型类型中。

  • 无界通配符
    有很多情况都和你在这里看到的情况类似，即编译器很少关心使用的是原生类型还是<?>。

  • 捕获转型
    有一种情况特别需要使用而不是原生类型。如果向一个使用的方法传递原生类型，
    那么对编译器来说，可能会推断出实际的类型参数，使得这个方法可以回转并调用另一个
    使用这个确切类型的方法。下面的示例演示了这种技术，它被称为捕获转型。

为什么需要RTTI

Run-Time Type Identification（RTTI）运行时类型信息识别

Class对象

类型信息在运行时的表示，是由Class对象完成的，它包含了与类有关的信息。Java使用Class对象来执行其RTTI。
所有的类都是在对其第一次使用时，动态加载到JVM中的。当程序第一次对类的静态成员进行引用时，
就会加载这个类。这个证明构造器也是类的静态方法。
为了生成类的对象，运行这个程序的Java虚拟机（JVM）将使用被称为“类加载器”的子系统。
类加载器首先检查这个类的Class对象是否已经加载。如果尚未加载，默认的类加载器就会
根据类名查找.class文件。在这个类的字节码被加载时，它们会接受验证，以确保其没有被
破坏，并且不包含不良代码（这是Java中用于安全防范目的措施之一）。

  • 获取Class对象的三种方法

当使用”.class”来创建对Class对象的引用时，不会自动地初始化该Class对象。
为了使用类而做的准备工作实际包含三个步骤：

  1. 加载。这是由类加载器执行的。该步骤将查找字节码，并从这些字节码创建一个Class对象。
  2. 链接。在链接阶段将验证类中的字节码，为静态域分配存储空间，并且如果必须的话，讲
    解析这个类创建的对其他类的所有引用。
  3. 初始化。如果该类具有超类，则对其初始化，执行静态初始化器和静态初始化块。
    初始化被延迟到了对静态方法或者非常数静态域进行首次引用时才执行。但是如果一个域被设置
    为static和final，但它需要执行其他操作才能获得初值，那就不会进行延迟初始化。

通过异常处理错误

概念

基本异常

当抛出异常后，有几件事会随之发生。首先，同Java中其他对象的创建一样，
将使用new在堆上创建异常对象。然后，当前的路径被终止，并且从当前环境
中弹出对异常对象的引用。此时，异常处理机制接管程序，并开始寻找一个
恰当的地方来继续执行程序。这个恰当的地方就是异常处理程序，它的任务
是将程序从错误状态中恢复，以使程序能要么换一种方式运行，要么继续运行下去。

  • 所有标准异常类都有两个构造器：一个是默认构造器；一个是接受字符串
    作为参数，以便能把相关信息放入异常对象的构造器。
  • 能够抛出任意类型的Throwable对象，它是异常类型的根类。

捕获异常

创建自定义异常

异常说明

  • 异常说明使用了附加的关键字throws，后面接一个所有潜在异常列表。
  • 代码必须与异常说明保持一致。
    如果方法里的代码产生了异常却没有进行处理，编译器会发现这问题
    并提醒你：要么处理这个异常，要么就在异常说明中表明此方法将产生异常。

捕获所有异常

  • void printStackTrace()
    void printStackTrace(PrintStream)
    void printStackTrace(java.io.PrintWriter)
    打印Throwable和Throwable的调用栈轨迹。
  • Throwable fillStackTrace() 用于在Throwable对象的内部记录栈帧的当前状态。
  • printStackTrace()方法提供的信息可以通过getStackTrace()方法来直接访问，
    这个方法返回一个栈轨迹中的元素所构成的数组，其中每一个元素都表示栈中的一帧。
    元素0是这个Throwable被创建和抛出之处。
  • Throwable的子类在构造器中都可以接受一个cause（因由）对象作为参数。这个cause
    就用来表示原始异常，这样通过把原始异常传递给新的异常，使得即使在当前位置创建并
    抛出异常，也能通过这个异常链追踪到异常最初发生的位置。

Java标准异常

Throwable这个Java类被用来表示任何可以作为异常被抛出的类。Throwable对象可分为
两种类型（指从Throwable继承而得到的类型）：Error用来表示编译时和系统错误（一般
不需要关心）；Exception是可以被抛出的基本类型。
RuntimeException是运行时异常，它们会自动被java虚拟机抛出，所以不必再异常说明
中把它们列出来。它们被称为“不受检查异常”，这种异常属于错误，将被自动捕获。

使用finally进行清理

  • 无论是否抛出异常，finally子句都会执行。（break、return，finally也会执行）
  • 异常丢失

异常的限制

当覆盖方法时，只能抛出在基类方法的异常说明里列出的那些异常。

构造器

异常匹配

  • 抛出异常时，异常处理系统会按照代码的书写顺序找出“最近”的处理程序。找到匹配的
    处理程序之后，它就认为异常得到处理，然后就不再继续寻找。
  • 查找的时候并不要求抛出的异常同处理程序所声明的异常完全匹配。派生类对象也可以
    匹配其基类的处理程序。

其他可选方式

异常使用指南

应该在下列情况下使用异常：

  1. 当恰当的级别处理问题。（在知道该如何处理的情况下才捕获一场）
  2. 解决问题并且重新调用产生异常的方法。
  3. 进行少许修补，然后绕过一场发生的地方继续执行。
  4. 用别的数据进行计算，以替代方法预计会返回的值。
  5. 把当前运行环境下能做的事情尽量做完，然后把相同的异常抛到更高层。
  6. 把当前运行环境下能做的事情尽量做完，然后把不同的异常抛到更高层。
  7. 终止程序。
  8. 进行简化。
  9. 让类库和程序更安全。

匿名内部类

如果定义一个匿名内部类，并且希望它使用一个在其外部定义的对象，
那么编译器会要求参数引用是final的。如果有匿名类的基类构造器传入参数，
该参数可以不为final。

嵌套类

  • 要创建嵌套类的对象，并不需要其外围类的对象。
  • 不能从嵌套类的对象中访问非静态的外围类对象。

包含抽象方法的类叫做抽象类，但也可以创建没有抽象方法的抽象类，抽象类中可以具有实现的方法。
static和abstract不能同时使用；

接口

interface只能使用默认或public访问权限修饰（除了嵌套接口），interface接口中，
方法被定义为是public的，但如果不使用public修饰接口，方法只能得到包访问权限。
不能实例化接口和抽象类。
JDK8后，interface接口可以有默认实现(“default void fun(){}”)。

继承多个接口，并且接口中含有方法名相同，但不能构成重载时，会发生错误。

再论向上转型

转机

将一个方法调用同一个方法主体关联起来称为绑定。若在程序执行前绑定，叫做前期绑定；
若在运行时根据对象的类型进行绑定，成为后期绑定（动态绑定或运行时绑定）。
Java中除了static方法和final方法（private方法属于final方法）之外，
其他方法都是动态绑定的。

构造器与多态

  • 基类的构造器总是在导出类的构造过程中被调用，而且按照继承层次逐渐向上链接，
    以使每个基类的构造器都得到调用。在导出类的构造器主体中，如果没有明确指定调用
    某个基类构造器，它就会默认调用基类默认无参构造器，如果不存在默认构造器，
    编译器会报错，则需要显式调用一个带参数的基类构造器。
  • 继承与清理
    当覆盖被继承的清理方法时，务必记住调用基类版本的清理方法，避免未清理基类垃圾。

协变返回类型

Java SE5中添加了协变返回类型，它表示在导出类中的被覆盖方法可以返回基类方法的
返回类型的导出类型，在java早期版本只能具有相同返回类型。

结合使用组合和继承

  • 确保正确清理
    首先，执行类的所有特定的清理动作，其顺序同生成顺序相反（通常这就要求基类
    元素仍旧存活）；然后，调用基类的清理方法。
  • 名称屏蔽
    如果Java的基类拥有某个已被多次重载的方法名称，那么在导出类中重新定义该方法名称
    并不会屏蔽在基类中的任何版本（这一点与C++不同）。
  • @Override注解
    添加@Override注解，表示覆写基类方法，而不是重载，如果基类没有该方法，则会报错。

向上转型

final关键字

  • final数据
    final标识基本类型，表示常量，不可修改值；
    final标识对象，引用对象不可变，但可以改变对象本身的值。
    空白final，指声明为final但不给出初值，那么必须在构造器中赋初值。
    final参数，意味着在方法中无法更改参数引用所指向的对象。
  • final方法
    final修饰方法，表示把方法锁住，以防任何继承类修改它的含义，确保在继承中是方法
    行为保持不变，并且不会被覆盖。但是，“覆盖”只有在某方法是基类的接口的一部分时
    才会出现。即，必须能够将一个对象向上转型为它的基本类型并调用相同方法。如果某方法为private，
    它就不是基类的接口的一部分。此时如果导出类具有相同名称但未public、protected或
    包范文权限方法的方法，则并不会覆盖该方法，仅是生成了一个新方法而已。
  • final类
    final修饰类，表示该类不能被继承。

初始化及类的加载

类的代码在初次使用时才被加载，这通常是指加载类的第一个对象之时，但是访问static域或static方法
时，也会发生加载。
